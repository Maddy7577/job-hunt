# Spec: Autohunt Mail Digest

## Overview
After an AutoHunt hunt completes, the user can click a **Send Digest** button to dispatch one personalised Gmail per job result. Each email contains the job's key details and a direct apply link, plus a role-specific PDF resume attached — generated by Google Gemini 1.5 Flash acting as a senior ATS screener. Gemini rewrites the user's primary resume to maximise keyword alignment with the job description: it re-ranks bullets, injects JD terminology, rewrites the summary, reorders skills, and enforces ATS-safe formatting — without fabricating any experience. The entire feature uses only free services: Gmail SMTP (app-password auth), Gemini 1.5 Flash free tier, and the `fpdf2` OSS library.

## Depends on
- **Step 01 — AutoHunt Setup**: `AutoHuntProfile` model, `/api/autohunt/hunt`, SSE stream, and the AutoHunt results table in the UI must all be complete and functional.

## Routes

| Method | Path | Description | Access |
|---|---|---|---|
| `POST` | `/api/autohunt/<int:search_id>/digest` | Start a digest run for the given AutoHunt search in a background thread. Returns `{ "status": "started", "total": N }`. | Public |
| `GET` | `/api/autohunt/<int:search_id>/digest/stream` | SSE stream — emits `digest_progress` events `{ "sent": N, "total": M, "current_company": "…", "current_title": "…" }` and a final `digest_done` event `{ "sent": N, "total": M, "errors": K }`. | Public |

## Database changes

New table `job_digest_status` — one row per job, tracks notification state:

```python
class JobDigestStatus(db.Model):
    __tablename__ = "job_digest_status"

    job_id      = db.Column(db.Integer, db.ForeignKey("job.id"), primary_key=True)
    status      = db.Column(db.Text, nullable=False, default="New")  # "New" | "Notified"
    notified_at = db.Column(db.DateTime, nullable=True)
```

- `job_id` — foreign key to `job.id`; uniquely identifies which open role this row tracks
- `status` — `"New"` on row creation; updated to `"Notified"` once the email is dispatched
- `notified_at` — UTC timestamp of dispatch; `NULL` while still `"New"`

A job with `status = "Notified"` is **never emailed again**, making the digest fully idempotent. No columns are added to the `job` table itself.

## Rules for implementation

- Use SQLAlchemy models; never write raw SQL.
- Use CSS variables — never hardcode hex values.
- Portal config belongs in `config/portals.json`, not in Python code.
- Mock mode must remain functional after the change (`APIFY_MOCK=true`). The digest feature is independent of Apify; it operates on already-stored `Job` rows, so mock mode is unaffected.
- No breaking changes to existing API response shapes.
- **Gemini prompt must be a single system+user call** — do not stream Gemini output; wait for the full tailored resume text before building the PDF.
- **Rate limiting**: enforce a minimum 4-second gap between consecutive Gemini API calls inside `send_digest()` to stay within the 15 RPM free-tier limit. Use `time.sleep(4)`.
- **No fabrication guard**: the Gemini system prompt must explicitly instruct the model not to invent job titles, companies, tools, metrics, or dates that are not present in the original resume text.
- **ATS-safe PDF**: the PDF must use single-column layout, standard fonts (Helvetica or Times), no tables, no text boxes, no images. Section headers must be plain text (`EXPERIENCE`, `SKILLS`, `EDUCATION`).
- **Filename sanitisation**: `Resume_<Company>_<Role>.pdf` — strip any characters outside `[a-zA-Z0-9 _-]` before forming the filename to prevent path traversal in attachments.
- **Digest is idempotent per job**: skip any job that already has a `JobDigestStatus` row with `status = "Notified"`. Only jobs with no row or `status = "New"` are processed on a second Send Digest click.
- **Graceful degradation**: if Gemini API key is not set, `send_digest()` must raise a clear `RuntimeError("GEMINI_API_KEY not configured")` caught at the route level and returned as `{ "error": "…" }` with HTTP 503 — never silently skip or send a blank resume.
- **SMTP errors per job**: a failed send for one job must not abort the whole digest loop. Log the error, increment an `errors` counter, and continue.
- **Email config route** (`GET /api/autohunt/email-config`) returns `{ "gmail_address": "…", "notify_email": "…", "configured": true/false }` — never expose the raw app password.

## Gemini ATS Prompt Contract

`digest_service.tailor_resume()` must send Gemini the following structured prompt (system + user):

**System message:**
```
You are a senior ATS screener and resume specialist with 15 years of experience.
Your task is to rewrite the provided resume so it passes ATS keyword matching for the given job description with the highest possible score.

Rules you must follow without exception:
1. Never invent, add, or imply experience, tools, job titles, companies, metrics, or dates that are not already present in the original resume.
2. You may rephrase existing bullet points using the job description's exact terminology where accurate.
3. Rewrite the professional summary to mirror the job title, required seniority, and top 3-5 must-have competencies from the JD.
4. Re-rank experience bullets within each role so the most JD-relevant achievements appear first.
5. Inject JD keywords into existing bullets only where they truthfully describe what the candidate already did.
6. Reorder the skills section: JD-matching skills first, then remaining skills, then deprioritised skills.
7. Remove tables, columns, text boxes, special Unicode characters, and non-ASCII bullets. Use only plain ASCII.
8. Use these exact section headers on their own line in ALLCAPS: SUMMARY, EXPERIENCE, SKILLS, EDUCATION, CERTIFICATIONS (omit any section not in the original).
9. Each job in EXPERIENCE must start with: ROLE TITLE | Company Name | Start Month Year - End Month Year (or Present)
10. Return only the rewritten resume text. No commentary, no explanations, no markdown.
```

**User message:**
```
=== ORIGINAL RESUME ===
{resume_text}

=== JOB DESCRIPTION ===
{job_description}
```

## Definition of done

- [ ] `POST /api/autohunt/<search_id>/digest` returns `{ "status": "started", "total": N }` where N equals the count of un-sent, non-filtered jobs in the search.
- [ ] SSE stream at `/api/autohunt/<search_id>/digest/stream` emits a `digest_progress` event for each job processed and a `digest_done` event when complete.
- [ ] Send Digest button in the AutoHunt tab is disabled until a hunt completes with ≥ 1 result, and re-enables after digest finishes.
- [ ] Progress bar updates correctly from 0% to 100% as emails are sent.
- [ ] Each sent job has a `JobDigestStatus` row with `status = "Notified"` and a populated `notified_at`; clicking Send Digest a second time sends 0 emails and shows "All jobs already sent".
- [ ] A tailored PDF resume named `Resume_<Company>_<Role>.pdf` is attached to each email (verify by opening the received email).
- [ ] The PDF is single-column, uses plain section headers (`SUMMARY`, `EXPERIENCE`, `SKILLS`, `EDUCATION`), and contains no tables or special characters.
- [ ] Email body contains: job title, company, portal, location, fit score, JD excerpt (≤300 chars), and a working apply link.
- [ ] `GET /api/autohunt/email-config` returns `{ "configured": true }` when all three env vars are set, and `{ "configured": false }` when any are missing — never exposes the app password.
- [ ] If `GEMINI_API_KEY` is not set, `POST /api/autohunt/<search_id>/digest` returns HTTP 503 with `{ "error": "GEMINI_API_KEY not configured" }`.
- [ ] If `GMAIL_APP_PASSWORD` is not set, route returns HTTP 503 with `{ "error": "GMAIL_APP_PASSWORD not configured" }`.
- [ ] Mock mode (`APIFY_MOCK=true`) continues to work normally for the hunt flow — digest is unaffected.
- [ ] Filenames with special characters in company or role names are sanitised (e.g. `Acme & Co` → `Resume_Acme__Co_Test_Manager.pdf`).
